### 1. Avoid Magic Strings

Do not hardcode role names or other important identifiers directly in the code, such as `"ADMIN"`. These are called *magic strings*. Magic strings create problems because:

* They are easy to mistype.
* They make refactoring difficult.
* They create inconsistency across the codebase.

Instead, define constants in a single location and reuse them.

Example:

```ts
export const ROLE = {
  ADMIN: "Admin",
  MEMBER: "Member",
  OWNER: "Owner"
} as const;
```

Then reference them:

```ts
if (user.role === ROLE.ADMIN) {
  ...
}
```

Benefits:

* Centralized definition
* Easier refactoring
* Autocomplete support
* Compile time safety

### 2. Understand `interface` vs `type`

TypeScript provides two ways to describe object shapes: `interface` and `type`.

Both can often achieve the same result, but they have different intended use cases.

**Use `interface` when:**

* Defining the shape of objects
* Modeling entities such as `User`, `RequestPayload`, etc.
* You want extensibility (interfaces can be merged and extended easily)

Example:

```ts
interface User {
  id: string;
  email: string;
}
```

**Use `type` when:**

* Creating unions or intersections
* Defining aliases for primitives or complex compositions

Example:

```ts
type Role = "Admin" | "Member" | "Owner";
```

General rule of thumb:
Use `interface` for object contracts and `type` for everything else.

### 3. Turn Off Auto Save While Debugging

When VS Code auto save is enabled, files get saved automatically during editing. This can cause:

* Repeated TypeScript recompilation
* Restart loops with `nodemon`
* Partial edits triggering runtime errors

During debugging or development with watchers, it is often better to disable auto save so that compilation happens only after deliberate saves.

### 4. Type Casting in TypeScript (`as` keyword)

Sometimes TypeScript cannot infer a type even when the developer knows what the type should be. In these cases type casting can be used.

Example:

```ts
const element = document.getElementById("input") as HTMLInputElement;
```

Here, TypeScript originally sees `HTMLElement | null`. Using `as` tells TypeScript the intended type.

However, casting should be used carefully because:

* It bypasses some type safety.
* Incorrect casts can introduce runtime errors.

Prefer proper typing and narrowing first. Use casting only when necessary.

### 5. Adding `user` to `req` in Authentication Middleware

When the authentication middleware attaches a `user` object to `req`, TypeScript does not automatically know that this field exists.

A common approach is globally modifying the Express `Request` type.

Example (global augmentation):

```ts
declare global {
  namespace Express {
    interface Request {
      user: User;
    }
  }
}
```

However, global modification has drawbacks:

* It assumes **every request has a user**.
* Some routes may not use authentication.
* It reduces accuracy of types.

A better approach is creating a **custom request type**.

Example:

```ts
interface AuthenticatedRequest extends Request {
  user: User;
}
```

Then use it only in authenticated controllers:

```ts
function controller(req: AuthenticatedRequest, res: Response) {
  ...
}
```

This keeps the type system precise and avoids misleading assumptions.

### 6. Specify Node Version in `package.json`

Different Node versions can cause inconsistent behavior across environments.

To avoid this, specify the required Node version in `package.json`.

Example:

```json
"engines": {
  "node": ">=18"
}
```

This helps:

* Developers install the correct version
* CI environments use compatible versions
* Avoid runtime incompatibilities

---

### 7. Using `.nvmrc` for Node Version Management

Another option is using **nvm (Node Version Manager)**.

Create a `.nvmrc` file in the project root:

```
18
```

Then developers can run:

```
nvm use
```

This automatically switches to the correct Node version for the project.

This approach is commonly used in many production repositories.

### 8. Clear Separation of Router, Controller, and Service Layers

A clean architecture improves maintainability and readability.

**Router Layer**

Responsibilities:

* Define API routes
* Attach middleware
* Forward requests to controllers

Example:

```ts
router.post("/orgs", authMiddleware, OrganizationController.create);
```

Routers should not contain business logic.

---

**Controller Layer**

Responsibilities:

* Receive request data
* Call services
* Format and return responses

Controllers contain **business rules**, but should not directly implement infrastructure logic.

Example:

```ts
const org = await OrganizationService.create(data);
res.json(org);
```

---

**Service Layer**

Responsibilities:

* Technical implementations
* Database queries
* External API calls

Services should not contain HTTP specific logic.

Example:

```ts
await prisma.organization.create({...});
```

This separation results in:

* Easier testing
* Clear responsibility boundaries
* Better scalability

### 9. What `prisma generate` Does

`prisma generate` creates the **Prisma Client**, which is a type safe database client generated from the Prisma schema.

Workflow:

1. Define models in `schema.prisma`
2. Run

```
npx prisma generate
```

3. Prisma generates:

* TypeScript types for models
* A database client (`PrismaClient`)
* Autocomplete for queries

Example:

```ts
const users = await prisma.user.findMany();
```

Without running `prisma generate`, the client and type definitions will not exist.

### 10. Why VS Code Reload Was Needed

After running `prisma generate`, new type definitions are created inside `node_modules`.

Sometimes the TypeScript language server in VS Code does not immediately detect new types.

Reloading the VS Code window forces:

* TypeScript server restart
* Reload of type definitions
* Correct IntelliSense

Command:

```
Developer: Reload Window
```

### 11. Understanding `typeof ROLE[keyof typeof ROLE]`

Given this object:

```ts
const ROLE = {
  ADMIN: "Admin",
  MEMBER: "Member",
  OWNER: "Owner"
} as const;
```

Breakdown:

**Step 1: `typeof ROLE`**

Gets the type of the object:

```
{
  ADMIN: "Admin";
  MEMBER: "Member";
  OWNER: "Owner";
}
```

---

**Step 2: `keyof typeof ROLE`**

Extracts the keys:

```
"ADMIN" | "MEMBER" | "OWNER"
```

---

**Step 3: `typeof ROLE[keyof typeof ROLE]`**

Indexes the object using all keys.

Result:

```
"Admin" | "Member" | "Owner"
```

This produces a **union of all values** inside the object.

Example usage:

```ts
type Role = typeof ROLE[keyof typeof ROLE];
```

Now:

```
Role = "Admin" | "Member" | "Owner"
```

Benefits:

* Values and types stay synchronized
* No duplication of role definitions
* Fully type safe

### Additional General Advice

1. Prefer strict TypeScript settings (`strict: true`).
2. Avoid `any` whenever possible.
3. Keep controllers thin and services reusable.
4. Validate request inputs (e.g., with Zod or Joi).
5. Write small focused functions rather than large controllers.

These practices significantly improve reliability and maintainability of a TypeScript backend codebase.
